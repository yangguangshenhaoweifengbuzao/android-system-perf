# Android 性能优化方法论

## 概述

性能问题可以分为三个维度：
1. **原有流程**（时序优化）：做这件事为什么这么慢？
2. **系统资源**（资源争夺）：谁在抢夺资源？为什么不够用？
3. **交互感知**（用户体验）：用户感觉慢，但数据为什么显示没问题？

---

## 一、原有流程方向（时序优化）

核心问题：**「做这件事为什么这么慢？」**

### 分析方法

#### Step 1: 建立时间线

画出完整的执行时间线，标记每个阶段：

```
────────────────────────────────────────────────────────────────────────────────────▶
│ A │ B │ C │ D │ E │ F │ G │ H │
```

#### Step 2: 识别瓶颈阶段

- 哪个阶段耗时最长？
- 哪个阶段是不必要的？
- 哪个阶段可以并行化？

#### Step 3: 优化策略（三板斧）

| 策略 | 说明 | 典型手法 |
|-----|------|---------|
| **删** | 减少不必要的步骤 | 启动时不必加载全部数据，布局中不必要的嵌套层级，重复的 measure/layout |
| **压** | 让每个步骤更快 | 用Protobuf替代JSON，异步加载图片，预编译布局 |
| **并** | 串行改并行 | 多线程并发加载资源，异步初始化SDK，DiffUtil计算与渲染并行 |

### 典型场景拆解

| 场景 | 时间线分析方法 | 常见瓶颈 |
|-----|---------------|---------|
| **应用启动** | ActivityThread.performLaunchActivity → onCreate → onStart → onResume | ClassLoader、资源加载、主线程阻塞 |
| **页面切换** | startActivity → onPause → onCreate → onStart → onResume → onStop | Intent解析、跨进程通信、布局渲染 |
| **列表滑动** | measure → layout → draw → SurfaceFlinger | 布局复杂度、overdraw、ViewHolder复用 |
| **网络请求** | DNS → TCP → TLS → Request → Response → Parse | DNS延迟、连接建立、JSON解析 |
| **IO操作** | open → read/write → close | 磁盘IO、内存拷贝、锁竞争 |

## 二、系统资源方向（资源争夺）

核心问题：**「谁在抢夺资源？为什么不够用？」**

### 资源问题的本质

资源问题的核心是**供需失衡**：
- **需求端**：应用/系统对资源的需求量
- **供给端**：系统可提供的资源容量
- **分配端**：资源如何在多个竞争者之间分配

当需求 > 供给，或分配不合理时，就会出现性能问题。

---
### 五大资源详解

#### 2.1 CPU 资源

**核心指标**

| 指标 | 含义 | 获取方式 |
|-----|------|---------|
| CPU使用率 | 总体负载 | `/proc/stat`、top |
| 线程CPU时间 | 每个线程占用 | `/proc/<pid>/task/<tid>/stat` |
| 调度延迟 | 等待CPU的时间 | perfetto sched slice |
| 运行队列长度 | 等待运行的线程数 | `/proc/loadavg` |
| CPU频率 | 当前运行频率 | `/sys/devices/system/cpu/cpu*/cpufreq/scaling_cur_freq` |

**分析方法**
- Step 1: 确认CPU是否是瓶颈  
- Step 2: 找出高CPU线程  
- Step 3: 分析CPU消耗原因 


**典型CPU问题及解决方案**

| 问题现象 | 可能原因 | 解决方案 |
|---------|---------|---------|
| 主线程CPU高 | 频繁GC | 减少对象分配，优化内存 |
| 主线程CPU高 | 复杂计算 | 移到子线程，算法优化 |
| 多线程CPU高 | 锁竞争 | 减少锁粒度，无锁设计 |
| 后台线程CPU高 | 忙等待 | 改用阻塞等待/条件变量 |
| 总体CPU高 | 线程过多 | 控制线程数量，用线程池 |

**CPU优化策略**

| 策略 | 说明 | 具体手段 |
|-----|------|---------|
| 减少需求 | 降低计算量 | 算法优化、缓存结果、懒加载 |
| 增加供给 | 提高计算能力 | 多核利用、绑核优化 |
| 调整分配 | 优化调度 | 线程优先级调整、任务队列管理 |
| 避开竞争 | 错峰计算 | 后台任务延迟、分片执行 |

---
#### 2.2 内存资源

**核心指标**

| 指标 | 含义 | 获取方式 |
|-----|------|---------|
| 进程内存 | PSS/ USS | `dumpsys meminfo <pid>` |
| 堆大小 | Java堆使用 | Memory Profiler |
| GC频率 | GC发生次数 | `logcat | grep GC` |
| GC暂停时间 | STW时长 | perfetto gc slice |
| 对象数量 | 各类对象数 | heapprofd、Memory Profiler |
| mmap区域 | 内存映射  | `/proc/<pid>/maps` |

**分析方法**

```
 Step 1: 确认内存是否是瓶颈
 Step 2: 分析内存分布     
 Step 3: 定位问题类型
 内存泄漏 → 对象持续增长，无法释放                                   
内存抖动 → 对象频繁创建销毁，触发频繁GC                              
大对象 → 单个对象占用过大                                            
过度分配 → 预分配或缓存过多 
```

**内存问题分类**

| 问题类型 | 特征 | 分析重点 |
|---------|------|---------|
| **内存泄漏** | 内存持续增长不回落 | 用Memory Profiler抓泄漏链 |
| **内存抖动** | GC频繁，内存锯齿状 | 找频繁创建的对象类型 |
| **OOM** | 单次分配过大或总量超限 | 看最大对象和总PSS |
| **内存压力** | 系统内存不足，LMK频繁 | 看系统可用内存 |

**内存优化策略**

| 策略 | 说明 | 具体手段 |
|-----|------|---------|
| 减少需求 | 降低内存占用 | 对象复用、数据压缩、按需加载 |
| 及时释放 | 避免泄漏 | WeakReference、生命周期管理 |
| 控制分配 | 减少抖动 | 对象池、预分配、避免onDraw中创建 |
| 分块处理 | 大对象拆分 | Bitmap分块加载、流式处理 |

#### 2.3 GPU 资源

**核心指标**

| 指标 | 含义 | 获取方式 |
|-----|------|---------|
| GPU使用率 | GPU负载 | `/sys/class/kgsl/kgsl-3d0/devfreq/load` |
| 渲染时间 | 每帧GPU耗时 | perfetto gpu slice |
| 过度绘制 | 重绘层数 | GPU Overdraw调试选项 |
| 帧缓冲 | Buffer占用 | `dumpsys SurfaceFlinger` |
| GPU内存 | 显存使用 | meminfo Graphics |

**分析方法**
- Step 1: 确认GPU是否是瓶颈
- Step 2: 分析过度绘制来源
- Step 3: 分析渲染管线

**GPU问题分类**

| 问题类型 | 特征 | 解决方向 |
|---------|------|---------|
| **过度绘制** | GPU负载高，耗电 | 减少重叠布局、移除背景 |
| **渲染慢** | 每帧GPU时间>8ms | 简化绘制、减少Shader复杂度 |
| **GPU饥饿** | GPU空闲但帧慢 | 看CPU端瓶颈 |
| **显存不足** | GPU OOM | 减小纹理、释放资源 |

**GPU优化策略**

| 策略 | 说明 | 具体手段 |
|-----|------|---------|
| 减少过度绘制 | 降低重绘 | 移除不必要背景、clipRect、View层级优化 |
| 简化绘制 | 降低GPU计算 | 减少渐变/阴影、简化Path |
| 纹理优化 | 减少显存 | 纹理压缩、按需加载 |
| 硬件层优化 | 合理使用 | 复杂静态View用硬件层，避免滥用 |

---
#### 2.4 IO 资源

**核心指标**

| 指标 | 含义 | 获取方式 |
|-----|------|---------|
| 读写延迟 | IO操作耗时 | perfetto fio slice |
| IOPS | 每秒IO次数 | `/proc/<pid>/io` |
| 读写量 | 数据量 | `dumpsys diskstats` |
| 队列深度 | 等待的IO请求 | `/sys/block/*/queue/depth` |
| IO等待 | CPU等待IO时间 | top wa列 |

**IO问题分类**

| 问题类型 | 特征 | 解决方向 |
|---------|------|---------|
| **主线程IO** | UI线程阻塞 | 移到子线程 |
| **随机IO多** | 大量小块读写 | 批量合并、顺序化 |
| **IO竞争** | 多进程同时读写 | 错峰、缓存 |
| **缓存失效** | 频繁读取相同数据 | 内存缓存 |
| **写入过多** | 频繁写盘 | 合并写入、延迟写 |

**IO优化策略**

| 策略 | 说明 | 具体手段 |
|-----|------|---------|
| 减少IO量 |降低读写次数 | 缓存、预加载 |
| 异步化 | 避免阻塞 | 子线程IO、AsyncTask |
| 批量化 | 合并小IO | BufferOutputStream、批量写入 |
| 选择合适存储 | 按场景选择 | SharedPreferences(少量)、SQLite(结构化)、文件(大量) |
| 优化SQLite | 数据库优化 | 索引、事务批量、预编译语句 |

---
#### 2.5 网络资源

**核心指标**

| 指标 | 含义 | 获取方式 |
|-----|------|---------|
| RTT | 往返时延 | ping、tcpdump |
| 带宽 | 吞吐量 | 网络测速 |
| 连接数 | 活动连接 | `/proc/net/tcp` |
| DNS延迟 | 解析时间 | perfetto dns slice |
| TLS握手 | 安全连接建立 | perfetto tls slice |

### 资源优化策略详解

| 策略 | 适用场景 | 具体手段 |
|-----|---------|---------|
| **减少需求** | 资源消耗过多，可优化 | 算法优化、缓存、对象复用、数据压缩 |
| **增加供给** | 资源容量不足，可扩展 | 多核利用、线程池扩展、内存预分配 |
| **调整分配** | 分配不合理，优先级错 | 线程优先级、任务队列、资源预留 |
| **避开竞争** | 多方竞争，时机不对 | 后台延迟、分片执行、错峰加载 |

### 资源问题速查表

| 症状 | 首查资源 | 分析工具 | 典型原因 |
|-----|---------|---------|---------|
| 滑动卡顿，帧率低 | CPU+GPU | perfetto、GPU调试 | 渲染复杂、过度绘制、主线程阻塞 |
| 应用响应慢，ANR | CPU | perfetto、traces.txt | 主线程阻塞、锁等待、GC暂停 |
| 内存持续增长 | 内存 | Memory Profiler | 内存泄漏 |
| 频繁GC，内存锯齿 | 内存 | Memory Profiler、logcat | 内存抖动、对象频繁创建 |
| 应用崩溃OOM | 内存 | dumpsys meminfo | 大对象、内存泄漏、总量超限 |
| 启动加载慢 | IO+网络 | perfetto | 文件读取、网络请求阻塞 |
| 后台耗电快 | CPU+网络 | Battery Historian | 后台轮询、唤醒频繁 |
| 网络请求慢 | 网络 | perfetto、tcpdump | DNS慢、连接建立慢、传输慢 |

## 三、交互感知方向（用户体验）

核心问题：**「用户感觉慢，但数据为什么显示没问题？」**

### 交互感知的本质

交互感知问题的核心是**「感知时间 ≠ 实际时间」**：
### 感知问题速查表

| 用户反馈 | 可能缺失 | 优化手段 |
|---------|---------|---------|
| 「点击没反应」 | 即时反馈缺失 | 涟漪效果、状态变化、触觉反馈 |
| 「不知道在加载」 | 进度指示缺失 | loading动画、进度条 |
| 「等了很久」 | 空白等待 | 骨架屏、占位图、渐进呈现 |
| 「跳转很突兀」 | 过渡动画缺失 | Shared Element Transition |
| 「不确定成功了」 | 完成反馈缺失 | 成功提示、状态确认 |
| 「感觉卡顿」 | 预加载不足 | 滑动预取、关联预取 |

## 五、问题分类速查表

| 症状表现 | 问题类型 | 分析切入点 |
|---------|---------|-----------|
| 启动慢 (>2s) | 流程问题 | ActivityThread trace，看main线程阻塞点 |
| 滑动卡顿 (>16ms/frame) | 流程+资源 | FrameInfo时间线 + GPU/CPU负载 |
| 内存OOM | 资源问题 | Heap分析，看对象分布和引用链 |
| 应用ANR | 流程问题 | traces.txt，看主线程等待点 |
| 网络慢 | 流程+资源 | 抓包看延迟分布，DNS/连接/传输 |
| 后台耗电 | 资源问题 | Battery Historian，看唤醒和CPU占用 |
| 交互「感觉慢」 | 感知问题 | 看首帧时间和视觉反馈时机 |

## 七、核心思想总结

- **流程问题**：画时间线，找瓶颈阶段，删压并
- **资源问题**：找瓶颈资源，找竞争者，减增调避
- **感知问题**：缩短首次反馈，渐进呈现，预判加载
